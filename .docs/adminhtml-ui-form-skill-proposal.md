## 1. Why вЂ” the gap and the recommendation

### The gap
The plugin has 17 skills built hub-and-spoke around `magento2-context`. Generation is covered for:

| Surface | Skill |
|---|---|
| Whole module scaffold (incl. a *basic* admin controller/UI stub) | `magento2-module-create` |
| Storefront (themes, RequireJS, Knockout, LESS, email) | `magento2-frontend-create` |
| EAV attributes (incl. form/grid *visibility* flags) | `magento2-eav-attribute` |
| GraphQL surfaces | `magento2-graphql-create` |
| Data migration / seeding | `magento2-data-migration` |

**Nothing owns the adminhtml UI-component *form* surface end-to-end.** `magento2-module-create`
emits an `admin-ui-*.php` stub and `magento2-frontend-create` is explicitly *storefront*. The thing
the team is actually struggling with вЂ” building a new admin **edit form** (form XML + `DataProvider`
+ `Modifier` + save/load controllers wired to a listing) вЂ” falls between those skills and is
generated by neither.

This is the single most boilerplate-heavy, most error-prone surface in Magento admin development:
the declarative UI-component form has ~6 moving parts that must agree on names, and a single
mismatch yields a blank form or a silent save failure with no useful error.

### Recommendation
**Build it.** It fills a real, named gap; it matches an existing, proven generator pattern in the
plugin (so low design risk); and it targets the team's stated pain point. Recommended name:
**`magento2-adminhtml-form`** (clearer than `magento2-adminhtml-ui`, and leaves room for a future
sibling `magento2-adminhtml-listing` for grids without overloading one skill).

> Forms-only now (per scope decision). A grid/listing generator is the natural Phase 2 and should be
> a **separate** skill that shares `references/` with this one вЂ” mirroring how `magento2-eav-attribute`
> and `magento2-data-migration` share TDD/coding-style references from `magento2-context`.

## 2. Skill identity (frontmatter)

Matches the house style: `name` + verb-first multi-line `description` with a "Use whenвЂ¦" trigger,
an explicit output summary, and a closing note on the review gate. No `allowed-tools` key (consistent
with every other skill in the plugin).

```yaml
---
name: magento2-adminhtml-form
description: |
    Generate a Magento 2 adminhtml UI-component edit form вЂ” the modern declarative
    ui_component/*_form.xml plus its DataProvider, Modifier (ModifierInterface/pool),
    and the save/load/back/delete controllers wired to a listing. Use when the user
    wants to add or scaffold a new admin form, an entity edit page, a "New/Edit"
    backend screen, fieldsets, dynamic-rows, file/image uploaders, a WYSIWYG field,
    or dependent/conditional fields in the Magento admin. Detects edition
    (Open Source vs Adobe Commerce) and notes Commerce-only form features. Produces
    files that pass magento2-module-review with zero Critical/High findings.
---
```

Trigger phrases deliberately include the words developers actually use ("admin form", "edit page",
"New/Edit screen", "fieldset", "dynamic rows", "uploader", "WYSIWYG") so the skill fires on natural
requests, not just the precise term "UI component".

## 4. Generator flow (SKILL.md body)

Follows the canonical phase flow used by `magento2-eav-attribute` / `magento2-graphql-create`.

### Phase 0 вЂ” Context Resolution
Invoke `magento2-context`; capture `{ctx}`. Use `{ctx.vendor}`, `{ctx.module_dir}`,
`{ctx.magento_cli}`, `{ctx.edition}`, `{ctx.runner}`, `{ctx.tools.xmllint}`, `{ctx.tools.phpcs}`,
`{ctx.tools.phpunit}`. **Never re-resolve.** Abort with a clear message if `{ctx.module_dir}` or the
target module is missing.

### Phase 1 вЂ” Resolve Inputs
Ask (via structured questions) for the values that can't be inferred:
- **Target module** (`{Vendor}_{Module}`) вЂ” must already exist; offer to call `magento2-module-create` if not.
- **Entity** name + how its data is loaded: existing model/repository/resource, or a plain
  `DataProvider` over a collection. (This is the #1 thing that determines the generated `DataProvider`.)
- **Primary field name** (the id field вЂ” drives data_source, persistor key, save redirect).
- **Fields**: for each вЂ” code, label, form element (input/select/wysiwyg/file/dynamicRows/вЂ¦),
  data type, required?, scope, default, source options, validation. A compact table input.
- **Surfaces wanted**: modifier? inline-edit? uploader? wysiwyg? dependent fields? delete?
  save-and-continue? menu entry? вЂ” each toggles templates on/off.
- **Listing it returns to** (for the back button / save redirect / insertListing) вЂ” name or "none yet".
- **ACL resource** for the controllers.

Look up domain guidance from `references/field-types.md` and `references/edition-differences.md`
while asking, so options offered match what the detected edition supports.

### Phase 2 вЂ” Plan (confirmation gate)
Present the file plan (every path to be created/modified, surfaces enabled, ACL/route/menu touches)
and **wait for "proceed"** вЂ” same gate `magento2-eav-attribute` uses before writing.

### Phase 3 вЂ” Test-first, then generate
- **3A (RED):** Write a failing test first, per the plugin's TDD discipline
  (`magento2-context/references/tdd-discipline.md`). For a form the highest-value cheap test is an
  **integration test** that boots the form's `DataProvider` and asserts `getData()` returns the
  expected shape for a known entity, plus a controller ACL/route test (the Save controller redirects
  + sets a success message; the Edit controller renders for an existing id and 404s/redirects for a
  missing one). Where a test DB isn't available, fall back to a unit test on the `Modifier`'s
  `modifyMeta`/`modifyData` and **record the gap** in the report (the established fallback pattern).
  Run and confirm it fails for the right reason.
- **3B (GREEN):** Generate from `templates/`, filling vendor/module/entity/field details. Write
  minimal code to turn the test green.

### Phase 4 вЂ” Verify
- `xmllint --noout` on every XML (`{ctx.tools.xmllint}`).
- `php -l` on every PHP file.
- `node --check` on `components.js`-style JS if any is emitted.
- Run the new test(s) with `{ctx.tools.phpunit}` в†’ confirm GREEN; run the module suite в†’ no regressions.
- `composer validate` if composer.json changed.
- Optionally invoke **`magento2-module-review`** on the touched module and confirm **zero Critical/High**
  (the shared acceptance gate every generator skill targets).

### Phase 5 вЂ” Report
Markdown report to `.docs/adminhtml-forms/{Vendor}_{Module}-{entity}-form-{date}.md`:
- files generated/modified (paths) + which surfaces were enabled
- test path + redв†’green evidence (or recorded gap)
- post-gen commands: `bin/magento setup:upgrade` (if di/routes/menu/acl changed),
  cache flush, `setup:di:compile` note for production
- the form's admin URL + ACL resource to grant
- links to `magento2-module-review`, and to `magento2-test-generate` for fuller coverage.

### 8.2 Form declaration & XML anatomy в†’ `references/form-xml-anatomy.md`
- **вњ… Verified (3-0):** A Form UI component is declared by a `<uiComponent name="{name}_form"/>` node in
  a **layout XML** file; its full configuration lives in a module **ui_component XML** file using a
  `<form>` root bound to the `urn:magento:module:Magento_Ui:etc/ui_configuration.xsd` schema. [S1]
  > *"The Form component is declared as a UI component node in layout files: `<uiComponent name="customer_form"/>`. Configuration occurs in module-level XML files at `view/base/ui_component/customer_form.xml` using the `<form>` root element with namespace `urn:magento:module:Magento_Ui:etc/ui_configuration.xsd`."* [S1]
- **рџ“„ Sourced:** ui_component XML files are **merged across modules**, so a module declares only its
  deltas, never a full copy вЂ” this is why the declarative approach replaces `Block\Widget\Form`. [S7]
- **рџ“„ Sourced:** Form structure = `fieldset` elements containing `field` elements, each configured via
  an `argument`/`item` data array (`label`, `sortOrder`, `collapsible`, `disabled`, вЂ¦). [S7]
- **рџ”№ Sourced:** Minimal new-form recipe (routes в†’ controller в†’ menu/acl в†’ `*_form.xml` в†’ DataProvider)
  corroborated by community walkthrough. [S17]

### 8.3 DataSource & DataProvider в†’ `references/dataprovider-patterns.md`
- **вњ… Verified (3-0):** The form's `DataSource` aggregates a class implementing
  `Magento\Framework\View\Element\UiComponent\DataProvider\DataProviderInterface`; the data it returns
  is **shared with all child components** of the form, keyed via `requestFieldName`/`primaryFieldName`. [S1]
  > *"DataSource aggregates an object of class implements the interface `\Magento\Framework\View\Element\UiComponent\DataProvider\DataProviderInterface` вЂ¦ Data provided by data source is shared and available for all components in the Assembly."* [S1]
- **рџ“„ Sourced:** Default backend DataProvider is
  `Magento\Framework\View\Element\UiComponent\DataProvider\DataProvider`, set via the `class` option;
  many custom forms instead **extend `Magento\Ui\DataProvider\AbstractDataProvider`**. [S4][S8]
- **рџ“„ Sourced:** `getData()` must return data shaped as `[entityId => [field => value, вЂ¦]]`; the
  `data-persistor` repopulates the form after a failed save. [S4][S10]
- **рџ”№ Sourced (real code):** Smile-SA seller module is the canonical pattern вЂ” DataProvider injects
  `PoolInterface`, and `getData()` iterates `getModifiersInstances()` calling `modifyData()`. [S8]

### 8.4 Modifier & Pool pattern в†’ `references/modifier-patterns.md`
- **рџ“„ Sourced:** A modifier implements `Magento\Ui\DataProvider\Modifier\ModifierInterface` with two
  methods вЂ” `modifyData(array $data)` (values) and `modifyMeta(array $meta)` (structure: name, label,
  type) вЂ” both accept and return arrays. May extend the Catalog `AbstractModifier`. [S3][S6]
- **рџ“„ Sourced:** Modifiers are registered as a **`Magento\Ui\DataProvider\Modifier\Pool` virtualType**
  in `etc/adminhtml/di.xml`; each entry has a `class` + numeric `sortOrder` for execution order. The
  pool is injected into the DataProvider's constructor. [S3][S6]
- **рџ“„ Sourced:** At runtime the modifier-produced structure is **merged** with the `form.xml`
  declaration вЂ” declarative XML and programmatic modifiers compose. [S3]
- **Design implication:** modifier is an **optional surface** вЂ” skip it for simple new forms; generate
  it (+`di-modifier-pool.xml`) only when extending an existing form or building dynamic fields.

### 8.9 Save/load flow, controllers & insert-form в†’ `references/controllers-and-routing.md`

### 8.11 Pitfalls / gotchas в†’ `references/pitfalls.md`
Synthesized from the core issue tracker [S16] + the failure modes above:
1. **Blank form** вЂ” `form.xml` `name`, the `<dataSource name>`, and the DataProvider `name` argument
   must all agree; a mismatch renders nothing.
2. **Silent save no-op** вЂ” `getData()` not keyed by entity id, or persistor/registry key mismatch.
3. **404 on Save/Delete** вЂ” missing `acl.xml` resource for the controller's `_isAllowed()`.
4. **Field renders but won't bind** вЂ” wrong `formElement` vs `dataType`, or missing `dataScope`.
5. **Uploader shows path not preview** вЂ” DataProvider didn't convert stored path в†’ `{name,url,...}`. [S12]
6. **Modifier runs in wrong order** вЂ” `sortOrder` collision in the pool. [S3]
7. **Dynamic-rows empty** вЂ” data supplied as assoc not indexed array. [S13][S14]
